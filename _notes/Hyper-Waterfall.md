오픈소스 프로젝트를 둘러보다 흥미로운 내용을 발견했다. 한국 문서파일인 .hwp 에디터를 Rust와 WebAssembly로 구현한 [rhwp](https://github.com/edwardkim/rhwp)의 [Hyper-Waterfall 철학](https://github.com/edwardkim/rhwp/blob/main/mydocs/manual/hyper_waterfall.md)이다. Claude code 등의 AI 사용을 전제로 Hyperfall, 에자일 개발론의 단점을 완화하고, 소규모 개발의 완성도를 높이는 방법이다.

앞으로 프로젝트에 해당 방법론을 사용해 구현하고자 하여,스스로 정리한 요약 내용을 블로그에 적어놓는다.

### 기존 방법론의 문제

rhwp 개발자 [edwardkim](https://github.com/edwardkim)은 워터폴의 철학을 긍정하면서도 계획, 문서화, 검증의 프로세스를 전부 사람이 하기에 속도가 너무 느렸다는 것을 지적한다. 에자일 역시 개발 속도를 빠르게 하기 위한 타협이라 이야기하며, 코드의 문서화와 스프린트마다 바뀌는 우선순위 등을 문제로 여긴다.
한편 AI 등장과 같이 생긴 바이브코딩에 관해서도 '극단적인' 시도로 일괄하며 이해하지 못하는 코드를 고칠 수 없다 이야기한다.

### 핵심 원리

제안한 해결책, Hyper-Waterfall은 AI를 사용해 **거시적으로는 워터폴, 미시적으로는 애자일** 개발을 동시에 진행하는 방법이다.

```
거시 (프로젝트 수준) — 워터폴의 규율:
  계획 ──→ 설계 ──→ 구현 ──→ 검증 ──→ 배포
  │         │        │        │        │
  ▼         ▼        ▼        ▼        ▼
  문서화    문서화    문서화    문서화    문서화

미시 (타스크 수준, 수 시간) — 애자일의 속도:
  구현 → 테스트 → 피드백 → 수정 → 테스트 → ... (빠른 반복)
  │       │         │        │       │
  AI     자동화    사람 판단   AI    자동화
```

거시적 방향 - 계획서, 승인, 단계별 보고, 최종 검증 - 은 워터폴의 규율로 통제하고, 미시적 실행은 에자일의 빠른 반복으로 수행하는 것이 요지다. [edwardkim]은 AI의 즉각적인 피드백 루프를 통해 전체 사이클이 수 시간 이내로 완료된다 이야기한다.

### 핵심 원칙

Hyper-Waterfall을 사용하기 위해 핵심적으로 지켜야할 사항은 아래와 같다

1. 구현 가치(목표)가 AI 컨텍스트에 유지되도록 한다
	- **CLAUDE.md** 에 프로젝트 규칙, 빌드 방법, 워크플로우를 명시
	- **타스크 등록 시** 목표와 범위를 명확히 기술
	- **컨텍스트가 길어지면** 핵심 맥락을 요약하여 재주입
2. 작업 통제권을 작업지시자가 유지해야만 한다
	- **계획은 AI가 제안하되, 승인은 사람이 한다**
	- **AI가 임의로 작업 종료를 제안하지 않는다** — 작업지시자가 결정
	- **아키텍처 변경은 반드시 사람의 판단을 거친다**
	- **"좋은 것 같습니다"로 넘어가지 않는다** — 직접 검증한다
3. 주기적으로 AI 컨텍스트 유지를 확인해야 한다
	- **컨텍스트 압축 시점** 을 인지한다 — AI가 이전 결정을 잊기 시작하면 경고 신호
	- **핵심 결정사항을 문서로 외부화** 한다 — AI 메모리가 아닌 파일에 기록
	- **새 세션 시작 시** 이전 맥락을 요약하여 전달한다
	- **메모리 시스템** 을 활용한다 — 프로젝트별 영구 기억 저장

### 역할 분담
개발을 위해 각자의 장점을 끌어올려 극대화하는 방법을 사용한다

사람은 **생각하는 역할**에 집중한다
- 방향 설정: "다음에 뭘 해야 하는가?"
- 우선순위: "무엇이 더 중요한가?"
- 품질 판단: "이것이 충분히 좋은가?"
- 아키텍처 결정: "이 구조가 올바른가?"
- 도메인 지식: "한컴은 이 경우 어떻게 동작하는가?"
- 피드백: "이 부분이 잘못되었다, 왜냐하면..."

AI는 **실행하는 역할** 에 집중한다:
- 분석: 코드베이스 탐색, 원인 추적
- 계획: 구현 계획서 작성
- 구현: 코드 작성, 테스트 생성
- 문서: 보고서, 기술 문서, 커밋 메시지
- 디버깅: 로그 분석, 수정안 제시
- 반복: 피드백 반영, 재시도

### 프로세스

Task Cycle은 다음과 같다.
```
1. 타스크 등록
   └─ 작업지시자: GitHub Issue 생성, 범위 정의
   
2. 수행 계획서
   └─ AI: 계획서 작성 (최소 3단계, 최대 6단계)
   └─ 작업지시자: 검토 → 승인 또는 수정 요청

3. 단계별 구현
   └─ AI: 코드 작성 + 테스트
   └─ AI: 단계 완료 보고서 작성
   └─ 작업지시자: 검증 → 승인 또는 피드백

4. 피드백 반영
   └─ 작업지시자: 피드백 문서 작성 (mydocs/feedback/)
   └─ AI: 피드백 반영, 수정

5. 최종 보고
   └─ AI: 최종 결과 보고서 작성
   └─ 작업지시자: 승인 → 이슈 클로즈
```

각각의 Task가 만들어 내는 산출물들은 다음과 같다. 해당 문서들은 AI가 작성하고, 사람은 검토/승인만 거치면 된다
```
mydocs/
├── orders/yyyymmdd.md          ← 오늘 할일 (타스크 목록 + 상태)
├── plans/task_{N}.md           ← 수행 계획서
├── plans/task_{N}_impl.md      ← 구현 계획서
├── working/task_{N}_step{M}.md ← 단계별 완료 보고서
├── working/task_{N}_final.md   ← 최종 결과 보고서
└── feedback/                   ← 코드 리뷰 피드백
```

또, 모든 Task는 다음 과정을 거쳐 품질을 보장받아야 한다.

1. **계획 승인** — 구현 전에 방향이 맞는지 확인
2. **테스트 통과** — cargo test (783+), cargo clippy
3. **시각적 검증** — SVG 렌더링 결과를 눈으로 확인
4. **E2E 검증** — 브라우저 렌더링까지 확인
5. **회귀 방지** — 기존 샘플 파일의 페이지 수 변동 없음

### 주의 사항

1. **도메인 전문성** — AI에게 올바른 방향을 제시하려면 도메인을 알아야 한다
2. **아키텍처 판단력** — AI의 설계 제안을 평가할 수 있어야 한다
3. **품질에 대한 기준** — "이 정도면 충분하다"를 판단할 수 있어야 한다
4. **AI 도구 숙련도** — AI의 강점과 한계를 알아야 한다
5. **규율** — 편하다고 바이브 코딩으로 빠지지 않는 자기 통제

다음 사항에 해당되는 프로젝트의 경우 적합하다.
- 복잡한 도메인 (역공학, 렌더링, 파싱 등)
- 높은 품질 기준 (상용 소프트웨어 수준)
- 문서화가 중요한 프로젝트
- 1인 또는 소규모 팀
